# Задача: Telegram-бот для humanize текстов (платный сервис для студентов)

## Что нужно построить

Telegram-бот на Python (aiogram 3.x), который принимает текст от пользователя, считает слова, показывает цену, ждёт оплату через Каспи (ручная верификация админом), и после подтверждения вызывает API сервиса ai-text-humanizer.com для обработки текста. Результат возвращается пользователю в чат.

---

## Бизнес-логика и цены

- **Цена за слово:** 0.75 ₸ (тенге)
- **Максимум слов за один запрос:** 5,000
- **Минимум слов за один запрос:** 50
- **Месячный лимит слов на API:** 50,000 (из них ~11,000 — личный расход владельца). Бот может использовать ~39,000 слов в месяц
- **Сумма округляется** до 2 знаков после точки (копейки): `round(words * 0.75, 2)` → например 375.00, 412.50, 1500.75
- **Валюта:** тенге (₸)

---

## Стек

- **Язык:** Python 3.11+
- **Фреймворк:** aiogram 3.x (асинхронный, FSM из коробки)

## FSM-состояния пользователя (states.py)

```
class OrderStates(StatesGroup):
    idle               # ожидание текста (начальное)
    text_received      # текст получен, цена показана, ждём подтверждения от юзера
    awaiting_payment   # юзер подтвердил, ждём скрин оплаты
    pending_approval   # скрин отправлен, ждём подтверждения от админа
    processing         # оплата подтверждена, идёт humanize через API
    done               # результат отправлен (автоматически переходит в idle)
```

## Основной флоу

### 1. /start
Приветственное сообщение:
```
Привет! Я бот для humanize текстов.

Отправь мне текст (от 50 до 5000 слов), и я рассчитаю стоимость обработки.

Цена: 0.75 ₸ за слово.
```

### 2. Пользователь отправляет текст (состояние idle)
- Принять текст (только текстовые сообщения, не файлы/стикеры)
- Подсчитать слова: `len(text.split())`
- Проверить лимиты:
  - Если < 50 слов → "Минимум 50 слов. Отправь текст побольше."
  - Если > 5000 слов → "Максимум 5000 слов за раз. Раздели текст на части."
  - Если слова превышают оставшийся месячный лимит → "К сожалению, лимит на этот месяц исчерпан. Попробуй в начале следующего месяца."
- Показать результат и спросить подтверждение:
  ```
  📝 Слов в тексте: 1,247
  💰 Стоимость: 935.25 ₸

  Хочешь продолжить? Отправь "Да" или "Отмена"
  ```
- Сохранить текст и количество слов во временное хранилище (FSM data)
- Перевести состояние → `text_received`

### 3. Подтверждение (состояние text_received)
- "Да" → Перевести в `awaiting_payment`, показать реквизиты:
  ```
  Переведи 935.25 ₸ на Каспи:
  📱 +7 XXX XXX XX XX (Имя Ф.)

  После оплаты отправь скриншот чека сюда.

  ⏰ У тебя 30 минут на оплату.
  ```
- "Отмена" → Сбросить в `idle`, сказать "Окей, отменено. Можешь отправить новый текст."

### 4. Приём скрина оплаты (состояние awaiting_payment)
- Принять ТОЛЬКО фото/документ (если текст — "Отправь скриншот чека")
- Форварднуть скрин админу (в личку) с информацией:
  ```
  🆕 Новый заказ #12
  👤 @username (ID: 123456789)
  📝 Слов: 1,247
  💰 Сумма: 935.25 ₸

  /approve_12 — подтвердить
  /reject_12 — отклонить
  ```
- Перевести в `pending_approval`
- Ответить юзеру: "Скрин получен! Ожидай подтверждения оплаты. Обычно это занимает несколько минут."

### 5. Админ подтверждает/отклоняет

**`/approve_{order_id}`:**
- Перевести юзера в `processing`
- Сообщить юзеру: "✅ Оплата подтверждена! Обрабатываю текст, подожди..."
- Вызвать API humanizer с сохранённым текстом
- Отправить результат юзеру (если текст длинный — файлом .txt)
- Записать потраченные слова в трекер
- Сбросить состояние в `idle`

**`/reject_{order_id}`:**
- Сообщить юзеру: "❌ Оплата не подтверждена. Свяжись с администратором, если считаешь что это ошибка."
- Сбросить в `idle`

### 6. Таймаут
- Если юзер в состоянии `awaiting_payment` или `pending_approval` больше 30 минут — автоматически сбросить в `idle`
- Сообщить юзеру: "⏰ Время на оплату истекло. Отправь текст заново, если хочешь повторить."
- Реализация: при создании заказа сохранять timestamp, проверять при каждом действии ИЛИ фоновой задачей (asyncio task каждые 5 минут)

## API-интеграция (services/humanizer_api.py)

Документация API: https://ai-text-humanizer.com/humanize-api/

- Метод: POST
- Эндпоинт: из документации (проверь актуальный URL)
- Заголовки: API ключ из конфига
- Тело: текст для humanize
- Ответ: обработанный текст

Обёртка должна:
- Делать запрос через aiohttp
- Обрабатывать ошибки (таймаут, 429 rate limit, 500)
- При ошибке API — уведомлять и юзера и админа
- Ретрай 2 раза с задержкой 5 секунд при 5xx ошибках

## База данных (SQLite)

### Таблица `users`
```sql
CREATE TABLE users (
    user_id INTEGER PRIMARY KEY,        -- Telegram user ID
    username TEXT,                       -- @username
    first_name TEXT,
    total_words_used INTEGER DEFAULT 0,  -- сколько слов всего использовал
    total_paid REAL DEFAULT 0,           -- сколько всего заплатил
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### Таблица `orders`
```sql
CREATE TABLE orders (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER REFERENCES users(user_id),
    text_hash TEXT,                      -- hash текста (не храним сам текст в базе)
    word_count INTEGER,
    price REAL,
    status TEXT DEFAULT 'pending',       -- pending, paid, completed, rejected, expired
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    completed_at TIMESTAMP
);
```

### Таблица `monthly_usage`
```sql
CREATE TABLE monthly_usage (
    month TEXT PRIMARY KEY,              -- формат "2025-01"
    words_used INTEGER DEFAULT 0,
    personal_words INTEGER DEFAULT 0     -- личный расход владельца (обновляется вручную)
);
```

## Трекинг слов (services/usage_tracker.py)

- При каждом успешном humanize — прибавлять word_count к `monthly_usage`
- Метод `get_remaining_words()` → 50000 - words_used - personal_words
- Метод `can_process(word_count)` → True если remaining >= word_count
- В начале каждого месяца — создавать новую запись (или проверять по текущему месяцу)

## Обработка ошибок

- **API humanizer вернул ошибку:** уведомить юзера "Произошла ошибка обработки, мы разберёмся. Деньги будут возвращены." + уведомить админа с деталями
- **Текст слишком длинный для одного API-запроса:** если API имеет свой лимит на длину — разбить на части, отправить последовательно, склеить результат
- **Юзер отправляет что-то не то в неправильном состоянии:** мягко перенаправить ("Сейчас я жду скрин оплаты. Отправь фото чека или напиши /cancel для отмены")
- **Все необработанные исключения:** ловить, логировать, отправлять юзеру "Что-то пошло не так, попробуй позже"

## Что НЕ нужно делать

- **Не нужны** очереди (BullMQ, Celery) — масштаб маленький
- **Не нужен** rate limiting — оплата сама по себе лимитирует
- **Не нужен** Redis — SQLite + FSM aiogram хватит
- **Не нужен** антифрод — админ проверяет каждый скрин вручную
- **Не нужна** автоматическая проверка скринов (OCR) — ненадёжно и легко обмануть

## Дополнительные мелочи

1. **Команда /cancel** — должна работать из любого состояния и сбрасывать в idle
2. **Если юзер отправляет новый текст в середине процесса** — напоминать что у него есть активный заказ
3. **Результат длиннее 4096 символов** (лимит Telegram) — отправлять как .txt файл
4. **Логирование** — все заказы, ошибки API, действия админа в файл и stdout
